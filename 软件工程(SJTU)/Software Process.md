# 软件过程模型

### 传统软件工程模型

1. #### 瀑布模型

   ![image-20210308201731475](C:\Users\Administrator\AppData\Roaming\Typora\typora-user-images\image-20210308201731475.png)

   - 以文档为驱动，每个阶段都具有与其关联的里程碑和产品
   - 每个阶段结束前完成文档审查，及早改正错误
   - 特点
     - 线性模型
     - 阶段间具有**依赖性**和**顺序性**
     - **推迟实现**
   - 带反馈的瀑布模型（当后面阶段发现前面阶段的错误，则沿反馈线返回并修正）
   - V模型![image-20210308202407057](C:\Users\Administrator\AppData\Roaming\Typora\typora-user-images\image-20210308202407057.png)
   - 缺点![image-20210308202158538](C:\Users\Administrator\AppData\Roaming\Typora\typora-user-images\image-20210308202158538.png)
   - 适用场合
     - 系统**需求明确且稳定**、**技术成熟**
     - 工程管理较严格的场合，如**军工、航天、医疗**

2. #### 原型模型

   - 目的
     - 明确并完善**需求**，演示原型
     - 研究**技术选择方案**，技术验证原型
   - 结果
     - 抛弃原型
     - 把原型发展成最终产品
   - 一个部分开发的产品，使客户和开发人员能够对计划开发的系统的相关方面进行检查。
   - ![image-20210308202708652](C:\Users\Administrator\AppData\Roaming\Typora\typora-user-images\image-20210308202708652.png)
   - 特点
     - **减少需求不明确带来的风险**
   - 缺点
     - 构造原型所用的技术和工具**不一定主流**
     - 快速建立起来的系统加上连续的修改可能导致原型**质量低下**
     - 设计者在**质量和原型中进行折中**
     - 客户意识不到一些**质量问题**
   - 适用场合
     - 客户定义一个**总体目标集**，但不清楚系统的**具体输入输出**
     - 开发者不确定**算法的效率**、软件与操作系统是否**兼容**以及客户与计算机交互的方式

3. #### 增量模型

   - 增量：满足用户需求的一个子集，能完成一定功能、小而可用的软件
   - ![image-20210308203116497](C:\Users\Administrator\AppData\Roaming\Typora\typora-user-images\image-20210308203116497.png)
   - ![image-20210308203143166](C:\Users\Administrator\AppData\Roaming\Typora\typora-user-images\image-20210308203143166.png)
   - 增量模型是一种非整体开发的模型，是一种进化式的开发过程
   - 增量模型从部分需求出发，先建立一个不完整的系统，通过测试运行这个系统取得经验和反馈，进一步使系统扩充和完善
   - 增量模型结合了原型模型的基本要素和迭代的特征
   - 每个增量的开发可用**瀑布**或**快速原型模型**
   - 特点
     - 增量的引入，**不需要提供完整的需求**，只要有一个增量出现，开发就可以进行。可以**更快投入市场**。
     - 在项目初始阶段**不需要投入太多人力资源**
     - 产品逐步交付，软件开发能够**较好适应需求的变化**
     - 能看到软件中间产品，提出改进意见，**减少开发风险**
     - 开放式体系结构，**便于维护**
   - 缺点
     - 每个增量必须提供一些系统功能，开发者很难根据客户需求给出**大小合适的增量**
     - 开放式体系结构很难构造
     - 易退化成边做边改的方式，使软件过程控制失去整体性
   - 适用场合
     - 软件开发中需求可能发生变化，具有较大风险的项目
     - 希望尽早进入市场的项目

4. #### 螺旋模型

   - 把**开发活动**和**风险管理**结合起来控制风险
   - ![image-20210308204606349](C:\Users\Administrator\AppData\Roaming\Typora\typora-user-images\image-20210308204606349.png)
   - 特点
     - 强调原型的**可扩充性**和**可修改性**，原型的进化贯穿整个软件生命周期，提高适应能力，支持用户需求的**动态变化**
     - 原型可看作可执行的需求的规格说明，易于为**用户和开发人员共同理解**，还可作为继续开发的基础，并为用户参与所有关键决策提供方便
     - 为项目管理人员**及时调整管理决策**提供了方便，进而可**降低开发风险**
   - 缺点
     - 如果每次迭代的效率不高，致使迭**代次数过多**，将会**增加成本**、**推迟交付时间**
     - 需要开发团队水平高，具有丰富的风险评估经验和专业知识，否则增加风险
   - 适用场合
     - 适用于**需求不明确**或者**需求可能发生变化**的**大型复杂的软件系统**
     - 支持**面向过程、面向对象**等多种软件开发方法， 是一种具有广阔前景的模型

5. #### 喷泉模型

   - 以**用户需求**为动力，以**对象为驱动**的模型，主要用于描述**面向对象**的软件开发过程
   - 软件开发早期**定义对象**，整个开发过程**充实和扩充对象**
   - 各个阶段使用**统一的概念和表示方法**，生命周期各阶段**无缝连接**
   - 各个开发步骤多次反复**迭代**
   - ![image-20210308233450366](C:\Users\Administrator\AppData\Roaming\Typora\typora-user-images\image-20210308233450366.png)
   - 特点
     - 喷泉模型的**各个阶段没有明显的界限**，开发人员可以**同步进行开发**， 可以提高软件项目开发效率，节省开发时间
     - 适应于**面向对象**的软件开发过程
   - 缺点
     - 由于喷泉模型在各个开发阶段是重叠的，在开发过程中需要大量的开发人员，因此**不利于项目的管理**
     - 喷泉模型要求**适用于面向对象开发**，使得审核的难度加大，尤其是面对可能随时加入的各种信息、需求与资料的情况
   - 适用场合
     - 适用于**面向对象开发**
